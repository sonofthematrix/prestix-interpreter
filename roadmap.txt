Fase 2 — Runtime Flow Stabiliseren
Doel
Alles wat mensen zeggen moet worden vastgelegd. Vertaling mag achterlopen, maar input mag nooit verloren gaan.
Definitieve flow
SpeechRecognition / typed input→ transcript buffer→ segmenter/logger→ conversation entry: pending→ translator queue→ status: translating→ output invullen→ status: translated/error→ speechSynthesis output
Belangrijke regel
Capture mag nooit wachten op translation.Translation mag achterlopen.
LIVE mode
Voor korte gesprekken.
- buffer 1000–1500 ms- snelle flush- snelle vertaling- korte zin per log-entry
STORY mode
Voor langere verhalen of monologen.
- langere buffer / delayed segmentation- alles blijven verzamelen- segment pas maken na langere pauze- vertaler mag achterlopen- geen input overslaan
Actie
In:
src/app/page.tsx
Zorgen dat:
- transcript buffer nooit wordt gewist bij recognition restart- queue niet blokkeert op speechSynthesis- speechSynthesis heeft timeout fallback- no-speech is non-fatal- second/third sentence wordt opgepakt
Acceptatietests
1. LIVE:   "Mereka tidak mengerti"   → snelle vertaling2. STORY:   "Saya mau pesan kamar untuk malam ini karena saya datang terlambat dan teman saya belum sampai"   → één compleet segment3. Doorpraten terwijl vorige vertaling loopt:   nieuwe zin moet pending entry worden4. no-speech:   geen rode fatal error

Fase 4 — Speaker Labeling
Probleem
Beide sprekers kunnen Indonesisch spreken. Taalherkenning kan dus niet bepalen wie wie is.
Oplossing
Geen automatische diarization in deze fase. Gebruik manual speaker labeling.
Features
- elke entry start als Unknown of actieve speaker- per entry buttons: Speaker A / Speaker B / Unknown- active speaker selector: Unknown / A / B- shortcuts:  Alt+1 = Speaker A  Alt+2 = Speaker B  Alt+0 = Unknown
Persistence
localStorage key:prestix-interpreter-conversation-log
Later
Echte automatische speaker diarization kan later, maar vereist audio-opname + diarization/transcription model. Niet betrouwbaar via browser SpeechRecognition alleen.
Acceptatie
- Speaker A/B label wijzigt zonder nieuwe vertaling- labels blijven na refresh- typed en speech entries worden gelogd

Fase 5 — Vertaalstijl: Natural Interpreter
Doel
De vertaling moet klinken als een menselijke live-tolk, niet als letterlijke woordenboekvertaling.
Promptregels
Voor id-en:
- output English only- interpreteer betekenis- vertaal slang natuurlijk- geen Indonesische normalisatie- geen uitleg- geen markdown
Voor en-id:
- output Indonesian only- natuurlijke gesproken Indonesische taal- niet te formeel tenzij input formeel is- geen uitleg
Slangregels
gua/gue/gw = I/melu/lo = younih = emphasis / vaak weglatendong = nadruk / soft insistenceaja = just/simply/only afhankelijk contextmampus = we're screwed / damn / dead afhankelijk contextajak = invite / bring along
Guards
- id-en output lijkt Indonesisch → retry stricter- en-id output lijkt Engels → retry stricter- output bijna gelijk aan input → retry natural interpreter
Tests
Jadi kalau gua ajak Wali mampus nih→ So if I invite Wali, we're screwed.Gua mau pesan kamar buat malam ini→ I want to book a room for tonight.Mereka tidak mengerti→ They don't understand.I want to book a room for tonight→ Saya mau pesan kamar untuk malam ini.

Fase 6 — Provider / Model Router
Correct principe
Gebruikers kiezen geen modellen. De app kiest intern het juiste model/provider.
Waarom
Je research-document bevestigt dat er geen universele beste vertaler is. De beste keuze hangt af van kwaliteit, latency, kosten, privacy, taalpaar en content-type. 
Interne rollen
LIVE_SHORT- korte live zinnen- lage latencySTORY_CONTEXT- langere context- natuurlijk interpreterenSLANG_CONTEXT- Indonesische straattaal- betekenis boven literalnessQUALITY_RETRY- als output te letterlijk/verkeerde taal isLOCAL_FALLBACK- als externe providers falen
Provider chain
Tokenizin alleen gebruiken als veilig gescopeerd.
Als Tokenizin translation-safe:1. Tokenizin2. Gemini3. DeepSeek4. OpenAI text5. local/OllamaAls Tokenizin niet safe/scoped:1. Gemini2. DeepSeek3. OpenAI text4. local/Ollama
De oude CODEXTRANSL-server had al multi-provider logica met OpenAI, DeepSeek en Gemini als concrete providers. 
Concrete modelrollen
LIVE_SHORT:- Gemini Flash / GEMINI_MODEL_FAST / gemini-2.0-flash- DeepSeek Chat- GPT-4.1 mini- localSTORY_CONTEXT:- Gemini 2.5 Pro / GEMINI_MODEL_PRO / GEMINI_MODEL_STORY- GPT-4.1- Gemini Flash- DeepSeek- localSLANG_CONTEXT:- Gemini 2.5 Pro- GPT-4.1- Gemini Flash- DeepSeekQUALITY_RETRY:- GPT-4.1- Gemini 2.5 Pro- GPT-4o / GPT-4 Turbo only if configuredLOCAL_FALLBACK:- Ollama/local configured model- later NLLB-200-style local model if added
Non-secret telemetry
Response mag tonen:
providermodelmodelRoletaskTypefallbackUsedfallbackChainTriedlearningMatchesCounttokenizinSkippedReason
Nooit tonen:
API keysheadersenv valuesauthorization
Acceptatie
- user-facing model picker absent- model routing internal- provider telemetry debug-only- Tokenizin skipped if not scoped

Fase 7 — Learning Module Uitbouwen
Huidige situatie
De learning module is geen echte modeltraining. Het is een correction / glossary / style memory. Dat is goed voor deze fase.
Opslag
data/interpreter-learning.json
Code
src/lib/interpreterLearning.tssrc/app/api/interpreter/feedback/route.tssrc/app/api/interpreter/route.ts
Uitbreiden naar drie memory-types
1. Correction memory
Voor volledige zinnen.
{  "type": "correction",  "sourceText": "...",  "wrongOutput": "...",  "correctedOutput": "...",  "mode": "id-en",  "note": "...",  "createdAt": "..."}
2. Glossary memory
Voor woorden/slang.
{  "type": "glossary",  "term": "gua",  "meaning": "I/me, informal Indonesian slang",  "mode": "id-en",  "examples": [],  "createdAt": "..."}
3. Style memory
Voor voorkeuren.
{  "type": "style",  "rule": "Translate naturally like a live interpreter, not literally.",  "mode": "id-en",  "createdAt": "..."}
Prompt-order
base interpreter rule→ mode rule→ learning correction/glossary/style context→ current input
De Prestix SSoT noemt prompt-composition order als belangrijk architectuurprincipe: base policy → mode instructions → persona/stage/KB/memory/checkpoint layers. Voor jouw app gebruiken we hetzelfde principe in mini-vorm. 
Acceptatie
- feedback route werkt- old correction-only data blijft compatibel- learningMatchesCount komt terug- slang test gebruikt glossary/correction
